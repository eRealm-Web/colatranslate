# 这不是一句 Prompt 生成的 App：可乐翻译是怎样被 AI 一步一步做出来的

很多人谈 AI 编程时，喜欢把过程描述成一句话：提需求，等结果，项目就出来了。

可乐翻译不是这样做成的。

这个项目真正的形成过程，更像是一场持续多天的工程协作：用户不断抬高要求、补充边界、纠正半成品倾向；AI 则不断从失败点切入，读代码、做假设、改一小步、跑验证，再继续下一轮。

如果要用一句话概括这次共创，我更愿意写成：

> 不是 AI 替人“自动生成了一个翻译 App”，而是人在不断给出更高标准，AI 在持续收敛实现路径，最后把一个最初带着大量 mock 和断点的 Flutter 工程，推进成了一个真正能本地运行的离线翻译项目。

这篇文章，就是把这段对话背后的工程过程整理出来。

## 起点：一个“看起来有下载模型”，但实际上跑不通的项目

这次对话最开始，并不是从“我要做一个全新的产品”开始的，而是从一个很具体的问题开始：

项目里已经有模型下载、设置页、翻译按钮这些表面功能，但运行时模型并没有被真正加载到应用里，原生层仍然是 mock，用户点“翻译”并不能得到真实的离线推理结果。

AI 做的第一件事不是大谈架构，而是从最具体的锚点入手：

- 读需求文档
- 读 Flutter 层的翻译状态和页面
- 查 MethodChannel 的调用链
- 对照 Android / iOS 原生实现

很快就发现了核心问题：

- `initEngine()` 是写死成功
- `listModels()` 是固定返回
- “下载模型”只是 UI 上有按钮，真实加载链路没有打通

这是整个项目第一次被“拉回现实”。

它也决定了后面这场协作的节奏：**先拆穿表象，再修真实链路。**

## 第一个分叉：用户明确拒绝“半成品”

当 AI 试图先把“下载后可见、状态可刷新”的链路打通时，用户马上把要求抬高了：

> “我是要你把整个写好，包括模型给我下载下来，然后打包到应用中，不用从线上下载下来……下次运行起来 app 是想要直接可以在输入框中输入文字后，点击翻译，就能翻译的，而不是一个半成品。”

这句反馈非常关键。

因为它把任务从“修补一个 demo”直接升级成了“交付一个真正可离线使用的移动端翻译应用”。

AI 接下来没有硬着头皮承诺“一次到位”，而是先去验证模型本身能不能在手机上跑：

- 查 Hugging Face 仓库实际文件
- 区分原始 safetensors 和移动端可用格式
- 评估移动端推理链路是否存在

这一步得出的结论很重要：

- 原始 `AngelSlim/Hy-MT1.5-1.8B-1.25bit` 仓库里的 `model.safetensors` 体积超过 4GB，不能直接用于手机端
- 真正适合移动端的是 GGUF 量化版本
- 想跑这个 1.25bit 模型，不能直接用主线 `llama.cpp`，必须依赖带 STQ1_0 支持的特定 PR 分支

换句话说，需求不是“把一个现成手机模型接进 Flutter”，而是“把一个特殊量化的大模型完整接入移动端本地推理链路”。

这也是整场协作的第二个关键点：**AI 不只是写代码，还要先证明哪条路是可行的。**

## 第二个分叉：没有退回简单方案，而是硬上 A 路线

面对这件事，其实有一条更容易的路：

- 换小模型
- 换云 API
- 或者改用系统翻译能力

AI 也如实把这些替代路线讲清楚了。但用户最后的选择非常明确：

> “A 路线搞起来”

这意味着项目正式走上最重的一条实现路径：

1. 下载 `Hy-MT1.5-1.8B-1.25bit.gguf`
2. 克隆 `llama.cpp`
3. 切到 `pr-22836-stq_0` 分支
4. 先在 macOS 上做推理 smoke test
5. 再分别打通 iOS / Android 的原生集成
6. 最后把 Flutter 和 FFI 链路完全接上

从这一步开始，这个项目不再是“Flutter UI 改造”，而是一个真正的跨平台原生推理工程。

## 真正把模型跑起来：从仓库拉源码，到本地验证推理

后面的动作就非常工程化了。

先是模型下载和源码准备：

- 下载 `Hy-MT1.5-1.8B-1.25bit.gguf`
- 克隆 `ggml-org/llama.cpp`
- 拉取并切换到 `pr-22836-stq_0`

然后不是直接往 App 里塞，而是先做最便宜、最可靠的验证：

- 先在 macOS 本地把 `llama.cpp` 构建起来
- 再用 `llama-completion` 对 GGUF 进行一次真实推理验证

这一步其实非常像资深工程师的工作方式：

如果底层模型和运行时本身都还没有被单独证明可用，就不应该急着把问题卷进 Flutter、Xcode、Gradle、MethodChannel、UI 状态这些更复杂的层里。

只有底层先通了，上层集成才有意义。

## 架构重构：从 MethodChannel mock 走向 Dart FFI + 原生 C ABI

真正把项目做“实”的关键，不只是把模型文件放进仓库，而是把调用链改成适合这个模型的架构。

这个过程中，项目发生了几次重要重构：

### 1. 翻译能力从假接口，变成真实 FFI 引擎

最初的 MethodChannel 更多是 UI 和原生 mock 的胶水。

后来逐步收敛为两部分职责：

- Flutter 通过 `dart:ffi` 直接调用 `cola_*` C ABI 做翻译
- MethodChannel 只负责模型路径解析和平台相关管理

这样做的好处是：

- 推理链路更直接
- 平台差异被压缩到最小
- Flutter 层可以把翻译当成一个明确的本地引擎来用，而不是一个黑盒平台方法

### 2. 翻译模型、STT、TTS 三条链路都改成端侧本地能力

项目后来不只是“文字翻译能跑”，而是把整个语音交互链条都本地化了：

- 翻译：`llama.cpp + Hy-MT GGUF + FFI`
- STT：`record + sherpa_onnx + SenseVoice-Small + Silero VAD`
- TTS：`supertonic_flutter + Supertonic 3 + ONNX Runtime`

尤其 STT 这块，最初系统语音识别是一个比较省事的方案，但后来被彻底替换成了本地 SenseVoice。

这一步非常有代表性：

一旦用户目标是“离线、可控、跨平台一致”，很多系统 API 提供的方便路径最终都会被替换掉。

### 3. UI 层也跟着架构进化

项目最后形成了三条主页面：

- 翻译页
- 语音翻译页
- 设备状态页

而语音翻译页的实时表现也不是简单把 partial 结果一路写进全局状态，而是专门保留页面局部状态，通过 `previewTranslate()` 做实时预翻译，避免每一个 partial 都污染全局输出和历史。

这说明 AI 在这个过程中做的不是“堆功能”，而是在持续修正抽象边界。

## 最大的坑之一：iOS Release 模式下的 FFI 符号丢失

如果只看功能实现，很多人会误以为最难的是把模型跑起来。

其实真实工程里，最费时间的往往是平台集成的边角坑。

这次最典型的坑，出现在 iOS Release 模式。

一开始，用户先提出了一个非常具体的现象：

> “你帮我把 iOS 的运行配置成 release 模式，不然每次运行起来之后断掉和控制台的连线 app 都会卡死。”

这一步先把 Xcode 的 shared scheme 改成了 Release LaunchAction，并关闭调试器附着。

但新的问题很快出现了：

> `Invalid argument(s): Failed to lookup symbol 'cola_init': dlsym(RTLD_Default, cola_init): symbol not found`

这说明在 Release 下，Dart FFI 要找的 `cola_init` 根本没有出现在最终 app 二进制里。

AI 的排查路径非常经典：

- 先确认 Dart 侧在 iOS / macOS 用的是 `DynamicLibrary.process()`
- 再确认 iOS 侧链接的是静态 XCFramework 切片
- 再检查 `OTHER_LDFLAGS` 里是否有 `-force_load`
- 最后直接对最终产物跑 `nm -gU`

结果发现：

- 静态库里的 `cola_*` 符号明明存在
- 但最终 `Runner.app/Runner` 里没有这些导出

根因不是“库没链接”，而是 **Release dead-strip 把这些运行时才会用到的 FFI 入口裁掉了**。

最后的解决办法很小，但很关键：

- 新增一个 `ios/Runner/ColaSymbols.m`
- 在里面显式强引用 `cola_init`、`cola_is_ready`、`cola_translate`、`cola_free_string`、`cola_shutdown`
- 把这个文件编入 Runner target

修完之后再跑 `nm -gU build/ios/iphoneos/Runner.app/Runner`，就能看到 `cola_*` 符号重新出现。

这类问题很能说明 AI 协作里真正有价值的部分：

不是“它知道答案”，而是**它能在复杂工程里把问题缩到一个可验证的局部，再通过一轮很便宜的验证把根因钉死。**

## 文档也不是附属品：旧文档被淘汰，新文档成为系统的一部分

当项目功能逐渐变实之后，用户又提了一个很多团队容易忽略、但实际上非常重要的要求：

把近期的所有技术细节和需求彻底整理进 `docs/`，而不是让 README 和零散注释各写各的。

这一步带来的变化并不只是“多了几篇 Markdown”，而是一次认知重建：

- 把旧的、已经不符合代码现实的文档删掉
- 把需求、系统结构、技术架构、平台构建说明分开
- 统一从 `docs/README.md` 进入

尤其重要的是，文档内容会随着架构变化同步更新。

比如当 STT 从系统 `speech_to_text` 切换到本地 SenseVoice 后，旧文档里那套描述就不能继续留着。这个动作本身，其实就是“把 AI 的中间产物变成项目长期资产”的过程。

## 资产治理：不是把大文件塞进 git，而是把工程变得可复现

到了后期，项目又遇到了另外一个非常现实的问题：

翻译 GGUF、SenseVoice ONNX、Supertone 模型和 voice style 文件体积都不小，不能一直粗暴地提交进仓库。

于是后面又做了一轮“资产外置化”改造：

- 在 `assets/` 下只保留目录结构和 README
- 大文件通过 `.gitignore` 排除出版本控制
- 根目录增加 `Makefile`
- 开发者通过 `make assets` 一次性恢复资源

这里最容易被低估的一点是：

AI 没有为了“快点完成任务”编造下载地址，而是去真实地查：

- Supertonic 的 Hugging Face 源
- sherpa-onnx 的 SenseVoice / Silero release 资源
- GGUF 镜像的有效下载地址

最后连根 README 也一起改掉了，确保新同事第一次拉代码就能按步骤运行。

如果说前半段是在做功能，这一段就是在做可维护性。

## 这次 AI 共创里，真正起作用的是什么

如果非要总结“AI 是怎么把项目一步步做成现在这个样子的”，我会说关键不在于模型有多聪明，而在于这几件事同时成立了：

### 1. 用户一直在抬高标准

这次对话里，最有价值的不是“给 AI 一个大而全的需求”，而是用户不断明确表达：

- 不要半成品
- 不要假下载
- 不要只在 Debug 下能跑
- 不要把大资产直接塞进 git
- 文档要完整，别人能跑起来

这些要求不断把项目从“演示态”往“交付态”推进。

### 2. AI 每次都从最具体的失败点切入

不是先抽象，不是先画大图，而是从这些锚点开始：

- 模型下载后没加载
- iOS Release 下符号找不到
- 文档过时
- 资产过大
- README 不够完整

这让每一轮改动都能快速验证，不容易失控。

### 3. 每次改动之后，都会马上验证

这次对话里反复出现的动作包括：

- `flutter analyze`
- `flutter build ios --release --no-codesign`
- `nm -gU`
- `xcodebuild -showBuildSettings`
- `llama-completion`
- `make -n assets`

AI 能真正帮上忙，不是因为它“能写很多代码”，而是因为它愿意不停验证，直到知道自己没有在骗自己。

### 4. 最终产出不只是代码，还有可交接的工程资产

这轮协作最后留下来的，不只是跑起来的功能，还有：

- 一套清晰的 `docs/`
- 一套可复现的 `Makefile`
- 一套可维护的资源目录说明
- 一套平台构建经验和坑位记录

这意味着项目不是“靠这次对话侥幸做成”，而是从一次对话里沉淀出长期可持续的结构。

## 项目现在到了什么位置

今天的可乐翻译，已经不再是一个只有页面壳子的 Flutter demo，而是一个具备以下能力的本地翻译项目：

- 离线文本翻译
- 本地 STT
- 本地 TTS
- 语音翻译页面的实时交互
- iOS / Android / macOS 的专项集成
- 文档、资产、构建流程的完整沉淀

当然，它仍然不是“全部结束了”。

真实项目永远还会继续：

- 更多平台适配
- 更稳的首启体验
- 更小的包体策略
- 更完整的测试覆盖
- 更多语言和交互细节

但至少现在，这个项目已经跨过了最关键的一道坎：

**它不再是一个“看起来差不多”的原型，而是一个能被持续推进的工程。**

## 最后想说的：AI 编程的价值，不在“替代开发者”，而在“放大工程推进速度”

如果把这次过程对外讲成“我用 AI 一句话做出了一个离线翻译 App”，那其实是把最有价值的部分讲丢了。

真正值得分享的是：

- AI 可以把代码搜索、根因定位、样板搭建、平台排错这些高频工作大幅加速
- 但前提是，人要持续给出清晰目标、持续纠错、持续抬高质量线
- 工程质量来自一轮又一轮小步验证，而不是来自一次完美 prompt

可乐翻译之所以能被一步步做成现在这个样子，不是因为 AI 会魔法，而是因为人和 AI 都在同一件事上持续用力：

**把每一个“差不多”都继续追问下去，直到它真的能用。**

---

如果你也在用 AI 做项目，我反而更建议你不要追求“让 AI 一次写完”，而是像这次一样：

1. 从一个具体问题开始。
2. 逼它面对真实约束。
3. 每次只推进一小步。
4. 改完立刻验证。
5. 把代码、文档、构建流程一起沉淀下来。

这样做，AI 才不是一个会说大话的聊天机器人，而会真正变成你的工程搭子。
