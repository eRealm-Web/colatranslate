# 可乐翻译需求说明

## 1. 产品定位

可乐翻译当前是一款以设备侧离线翻译为核心的轻量工具型应用。它的设计原则不是功能堆叠，而是把一条真正能跑通的离线翻译链路做短、做稳、做清楚。

核心原则如下：

- 翻译能力必须在本地完成，不接云端翻译 API。
- 语音识别和语音合成同样以本地执行为原则，不上传用户内容。
- 页面数量尽量少，操作路径尽量短。
- 不保留“看起来像功能、实际上没接好”的按钮和入口。
- 文案与交互优先强调可执行性，而不是概念化表达。

## 2. 目标用户与典型场景

### 2.1 目标用户

- 需要短句快速翻译的普通用户。
- 需要在网络不稳定或不希望上传文本时进行翻译的用户。
- 需要在面对面场景中做简短跨语言沟通的用户。

### 2.2 典型场景

- 文本输入后立刻得到译文，并可复制或发音。
- 通过本地语音识别把说话内容实时转成文字，再持续翻译。
- 检查本地模型和本地引擎状态，确认设备是否可用。

## 3. 当前版本功能范围

### 3.1 文本翻译页

目标：提供最直接的文本离线翻译路径。

需求：

- 用户可以选择源语言和目标语言。
- 源语言支持 `auto` 自动检测。
- 目标语言当前限定为 `zh / en / ja / ko / fr`。
- 用户可以手动输入文本，也可以点击语音输入按钮，把语音识别结果填入文本框。
- 输入内容上限为 256 字符。
- 点击“开始离线翻译”后，应用调用本地翻译引擎返回译文。
- 结果区必须支持复制。
- 结果区在目标语言受支持时必须支持本地 TTS 发音。
- 当用户点击空白处或拖动页面时，键盘应能收起。

验收标准：

1. 引擎可用时，输入短文本能稳定得到译文。
2. 源语言为 `auto` 时，文本翻译可以直接执行。
3. 语音输入前如果源语言仍是 `auto`，界面必须提示用户先选择具体语言。
4. 译文非空时，复制操作能够成功。
5. 目标语言在当前 TTS 支持范围内时，点击“发音”可以直接播报译文。

### 3.2 语音翻译页

目标：提供短句级、实时反馈的语音翻译工作流。

需求：

- 页面顶部只保留源语言、目标语言、交换按钮。
- 源语言当前只允许 `zh / en / ja / ko`，因为 STT 仅支持这些输入语言。
- 页面下半部分实时显示原始识别文本。
- 页面上半部分实时显示针对当前识别文本的译文。
- 底部只保留一个开始 / 停止按钮，用作 toggle。
- 开始后立即启动语音识别。
- 识别局部结果到来后，要做去抖翻译，避免每个碎片都触发一次重算。
- 识别自然结束或用户手动停止时，必须补做最后一次翻译，保留最后一段原文和最后一版译文。
- 在识别中、启动中、最终收尾翻译中，语言设置区域必须锁定，避免方向切换导致结果错乱。

验收标准：

1. 麦克风权限正常且 STT 资产就绪时，开始 / 停止按钮可以正常切换。
2. 识别期间原文区持续更新，译文区按节奏持续更新。
3. 停止后保留最后识别文本和最后译文，不会立即清空。
4. 页面不包含旧版的提交按钮、结果卡片、自动连续对话、双区长按对讲、页面内 TTS 等交互。
5. 目标语言不能切到与源语言相同；交换语言时需要校验 STT 是否支持新的输入语言。

### 3.3 设备状态页

目标：只展示当前版本真正有价值的设备和模型信息。

需求：

- 展示翻译引擎是否就绪。
- 展示当前错误信息。
- 提供重新检测按钮。
- 展示本机可见模型列表。
- 展示当前隐私说明。

验收标准：

1. 首次进入页面后会自动检测一次状态。
2. 点击“重新检测设备状态”会重新初始化引擎并刷新模型列表。
3. 页面不再暴露模型下载、模型更新、模型删除等管理入口。

## 4. 业务流程要求

### 4.1 文本翻译业务流程

1. 页面初始化时尝试预热本地引擎。
2. 用户输入文本或通过语音输入填充文本框。
3. 用户点击翻译按钮。
4. 控制器检查引擎是否就绪，必要时先做初始化。
5. 翻译任务进入后台 isolate 调用 FFI。
6. 译文返回后写入结果区，并加入内存历史。
7. 用户可对结果执行复制或发音。

### 4.2 语音翻译业务流程

1. 用户选择明确的源语言和目标语言。
2. 点击开始按钮，启动录音流、VAD 和本地识别器。
3. 识别结果持续输出到页面局部状态。
4. 页面局部状态触发去抖翻译。
5. 译文持续刷新，但不把局部结果写入全局历史。
6. 用户停止或识别自然结束时，执行最后一次收尾翻译。

### 4.3 设备状态业务流程

1. 页面加载后主动请求刷新。
2. 调用翻译控制器初始化引擎。
3. 通过 MethodChannel 获取本机模型列表。
4. 在页面上展示当前可见状态与错误信息。

## 5. 语言与能力范围

### 5.1 当前 UI 支持语言

- 翻译 UI：`auto / zh / en / ja / ko / fr`
- STT 输入语言：`zh / en / ja / ko`
- TTS 输出语言：`zh / en / ja / ko / fr`

### 5.2 当前模型能力边界

- 当前内置翻译模型为 `Hy-MT1.5-1.8B-1.25bit.gguf`。
- 翻译链路围绕短句 / 短段落场景设计。
- 当前工程的原生 prompt 中预留了更多语言映射，但 Flutter UI 没有全部开放。

## 6. 全局交互与体验要求

- 底部导航固定为三页：翻译、语音、设备。
- 视觉风格保持浅底、白色卡片、科技感冷色系渐变。
- 所有主要按钮文案必须清晰直接。
- 失败场景必须有可理解提示，例如：权限不足、模型找不到、引擎未就绪、翻译失败、语音初始化失败。
- 所有主流程都应有显式反馈，例如准备中、翻译中、合成中。

## 7. 数据、隐私与安全边界

- 用户输入文本、语音识别结果、译文不上传到云端服务。
- 翻译、语音识别、语音合成都在设备本地执行。
- 历史记录只保存在内存中，当前不持久化。
- 模型和语音资产作为应用资源或应用支持目录中的本地文件存在。

## 8. 明确不做

当前版本明确不包含以下内容：

- 收藏译文
- 历史记录持久化
- 自动连续对话模式
- 自动判断发言人
- 双区长按对讲 UI
- 模型下载中心
- 模型在线更新与删除管理页
- Android 悬浮窗取词
- 语音翻译页内的自动播报与重播控制

## 9. 平台与技术约束

- Flutter / Dart 为主实现栈。
- 状态管理使用 Riverpod。
- 翻译必须走 `dart:ffi` 调用原生 `cola_*` C ABI。
- iOS 最低版本为 16.0。
- macOS 最低版本为 14.0。
- Android 当前仅支持 `arm64-v8a`。
- GGUF 模型需要以真实文件路径提供给原生引擎，不能直接从 Flutter 资产内存流使用。

## 10. 后续候选方向

以下内容可以作为后续版本的候选规划，但不属于当前交付范围：

- 历史记录持久化与检索
- 更细粒度的模型状态展示
- 首启隐私确认流程
- 更完整的多语言 UI 范围
- 语音翻译页的对话记录导出
- Android 与 iOS 真机上更细的性能和耗时埋点
